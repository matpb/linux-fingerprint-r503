# Crypto-posture review — r503-fpr v2 authenticated wire protocol

**Date:** 2026-05-28
**Scope:** SPEC §13 implementation on `main` at branch point
`feat/crypto-posture-upgrade` (parent commit `837fd35`).
**Subject:** firmware (`firmware/r503fp/`, `firmware/r503fp_wipe/`),
daemon (`pcside/daemon/src/`), and the TPM2-seal path (SPEC §13.12).
**Reviewer:** project author (Mat) via multi-agent review.

## What this is — and is not

This is **not** a third-party formal cryptographic audit by a paid security
firm. It is a documented multi-pass review of the protocol, the
implementation, and the deployment posture, performed across the layers
SPEC §13 covers. Every finding below has a fix shipped on the same
branch; every claim is backed by code paths, tests, or runtime evidence
that a reader can re-execute.

The goal of writing it down: substitute concrete, reproducible evidence
for the previous "no formal cryptographic review" disclaimer in
SPEC §13 / README. Naming what was done makes "what was not done"
specific instead of vague.

## Methodology

Five layers reviewed independently before synthesis:

1. **Crypto primitive** — `firmware/r503fp/siphash.h`,
   `pcside/daemon/src/crypto.rs`. SipHash-2-4. KAT vectors and
   constant-time properties.
2. **Wire framing** — `firmware/r503fp/framing.h`,
   `pcside/daemon/src/framing.rs`. Frame parse / encode, MAC verify,
   domain separation, parser strictness.
3. **EEPROM counter ring** — `firmware/r503fp/eeprom.h`. Wear levelling,
   CRC-16, replay-counter persistence ordering.
4. **TPM2 seal** — `pcside/daemon/src/tpm.rs`. PCR policy, sealed blob
   parser, unseal failure mode.
5. **Daemon lifecycle** — `pcside/daemon/src/{sensor,sensor_actor,
   keystore,pairing,main,state}.rs`. Key handling in memory, pairing
   flow, sensor reopen, counter persistence ordering.

Tooling invoked:

- Direct code reading by the reviewer.
- Two independent Explore-agent passes (firmware + daemon halves) for
  cross-checking — the agent does not see the reviewer's notes and re-
  finds (or fails to re-find) the same issues.
- `cargo test` (49 unit + lib tests).
- Cross-implementation property test against the third-party
  `siphasher` crate (`tests/crossimpl_siphash.rs`, 1024 random vectors).
- Stable-Rust property fuzzer on the wire parsers
  (`tests/fuzz_framing_smoke.rs`, ~135 000 random / mutated inputs).
- `cargo audit` (RustSec advisory database).
- Firmware compile under `arduino-cli` to confirm strict-parser and KAT
  self-test changes fit on the ATmega328P (target: <80 % flash).

## Findings

Severity scale: **P0** critical (currently broken), **P1** real bug,
**P2** real bug but limited blast radius, **P3** spec hygiene / defence
in depth.

| # | Sev | Area | Issue | Fix | Status |
|---|---|---|---|---|---|
|  1 | **P1** | daemon framing | `if diff != 0` after XOR was *intended* constant-time but rustc gives no codegen guarantee. | Use `subtle::ConstantTimeEq` on both `verify_command` and `verify_response`. | **Fixed** — `pcside/daemon/src/framing.rs`. |
|  2 | **P2** | daemon key handling | SipHash key held in `R503::key`, `auth_key`, plus hex form, none of which scrub on drop. RAM / swap / core-dump leak. | All key surfaces wrapped in `zeroize::Zeroizing<...>` (16 B array form *and* `String` hex form). | **Fixed** — `keystore.rs`, `sensor.rs`, `sensor_actor.rs`, `tpm.rs`. |
|  3 | **P1** | daemon wire parser | `parse_response` / `verify_command` `split_at(s.len() - 19)` could panic if `s.len() - 19` landed inside a multi-byte UTF-8 character (lossy conversion of hostile bytes from the serial line). DOS vector. | Check `s.is_char_boundary(...)` before splitting; reject as `MissingMacSuffix` otherwise. | **Fixed** — `framing.rs::split_mac_suffix`. **Found by `tests/fuzz_framing_smoke.rs` in this review.** |
|  4 | **P2** | parser hygiene | Decimal token parsing accepted leading zeros (`"042"` → 42). Re-format-for-MAC step on both sides re-produces `"42"`, so a captured `C 42 ... M xyz` could be reshaped to `C 042 ... M xyz` and still validate. Replay-counter check blocks the real attack downstream; removing the surface is defence in depth. | Reject leading zeros (except exact `"0"`) on both the daemon (`parse_strict_u64`) and firmware (`parse_u64`) sides. | **Fixed** — `framing.rs`, `firmware/r503fp/framing.h`. |
|  5 | **P3** | daemon key generation | `keystore::generate_key()` opened `/dev/urandom` directly. SPEC §13.2 named `getrandom(2)`; the spec was the more correct primitive (always-blocking-until-seeded). | Use the `getrandom` crate. | **Fixed** — `keystore.rs`. |
|  6 | **P3** | daemon TPM blob | Sealed-blob deserializer used raw `+` on `usize` derived from a u32 length field, with no size cap. Hostile blob with 4 GB pub_len could try to preallocate. | `checked_add` + 64 KB per-field cap. | **Fixed** — `tpm.rs::deserialize_blob`. |
|  7 | **P3** | firmware wipe | `r503fp_wipe.ino` used `EEPROM.update()` which short-circuits cells already at `0xFF`. Power-loss mid-wipe could plausibly clear the key region but leave the `R503FPv2` magic intact. | Switch to `EEPROM.write()` (unconditional). | **Fixed** — `firmware/r503fp_wipe/r503fp_wipe.ino`. |
|  8 | **P3** | daemon key disposal | `delete_key` just `remove_file`s the plaintext key; underlying blocks persist. | Overwrite with `[0xFF; 33]` + `sync_all` before `remove`. Best-effort; full guarantee needs LUKS / at-rest encryption underneath, noted in comment. | **Fixed** — `keystore.rs::shred_file`. |
|  9 | **P3** | firmware boot | No runtime check that the SipHash primitive on-device matches its published vectors. A bad flash / compiler regression would silently accept (because MAC compute and MAC verify both use the same broken impl). | Boot-time KAT self-test (4 vectors). On any mismatch: halt with 6 Hz LED strobe + `FATAL siphash_kat_fail` serial line. ~120 bytes flash. | **Fixed** — `firmware/r503fp/r503fp.ino::siphash_kat_ok`. |
| 10 | **P3** | TPM seal flexibility | PCR7-only binding hard-coded. Operators who want to bind kernel/initrd updates (PCR11 via UKI) had no path. | New CLI flag `--seal-tpm-pcrs=<list>` (default `7`). On-disk format bumps to `R503TPM\x02` when used; existing `R503TPM\x01` blobs continue to load. Encoded PCR list is read back at unseal so the policy reconstructs automatically. | **Fixed** — `tpm.rs`, `keystore.rs::save_key_sealed_with_pcrs`, `main.rs`, `pairing.rs`, `dist/reseal-tpm.sh`. |

### Reviewed and confirmed sound (no fix needed)

These items were checked and verified to behave per SPEC. No
implementation change. They are listed so a re-reviewer knows they
were not skipped:

- **Counter persistence order** (`sensor.rs:312-327`). Persists
  `next_cmd_counter+1` BEFORE writing the frame on the wire. Crash
  windows lose a counter slot, never replay one. SPEC §13.4.
- **Strict `>` replay check on the firmware side**
  (`r503fp.ino` line ~313). `ctr <= last_seen` rejects, not `<`.
- **`save_counter` before handler dispatch**. Frame is committed to
  EEPROM before the side-effecting handler runs.
- **MAC failure has zero EEPROM side-effects.** Unframed `ERR
  mac_invalid` is the only observable change.
- **`pair` rejected on a paired Nano**, `unpair` framed-only on a
  paired Nano. Confirmed at `r503fp.ino::handlePair / handleUnpairFramed`.
- **`state.json` written atomically with `fsync`**. Confirmed at
  `state.rs::save`.
- **TPM unseal failure refuses to start with no plaintext fallback**.
  Confirmed at `main.rs::main_pre_open` and the `load_key_with_source`
  comment.
- **Domain separation in MAC inputs holds across cmd / resp**. Tested
  in `framing.rs::tests::domain_separation_holds` and
  `cmd_frame_not_accepted_as_response`.
- **KAT vectors agree with Aumasson reference**
  (`crypto.rs::tests::vector_len_{0,1,8,15}`).

## Cross-implementation evidence

`tests/crossimpl_siphash.rs` runs **1024 random (key, msg) vectors**
through both the in-tree hand-rolled `crypto::siphash24` and the third-
party `siphasher::sip::SipHasher24` crate. Both must produce bit-for-bit
identical MACs. Result on this branch:

```
running 2 tests
matches_siphasher_on_canonical_aumasson_vectors ... ok
matches_siphasher_on_1024_random_vectors ........ ok
test result: ok. 2 passed; 0 failed
```

`siphasher` is the descendant of Rust's `std::hash::SipHasher` (was
SipHash-2-4 until deprecation in 1.13), MIT-licensed, pulled in
transitively by serde-rs and millions of downstream crates. It is an
independent implementation — neither side referenced the other while
both were written.

## Fuzz evidence

`tests/fuzz_framing_smoke.rs` runs the four wire parsers
(`parse_command`, `parse_response`, `verify_command`, `verify_response`)
plus `crypto::siphash24` against ~135 000 inputs per CI run:

- 50 000 random byte sequences (any length 0..256, biased toward the
  framing minimum).
- 30 000 random ASCII-printable sequences.
- 60 000 mutations of valid frames (single-byte flip, truncation,
  splice-insert, byte-zero).
- 20 000 random-key + random-msg SipHash inputs.
- 10 000 + 10 000 encode → verify round-trip checks on random inputs.

Invariant: **must not panic**. **Found one panic** during this review
(finding #3 above), now fixed. Result on this branch:

```
running 6 tests
test result: ok. 6 passed; 0 failed
```

Long-corpus libFuzzer scaffolding lives at `pcside/daemon/fuzz/` for
operators with a nightly toolchain:

```
rustup toolchain install nightly
cargo install cargo-fuzz
cargo +nightly fuzz run framing -- -max_total_time=3600
```

## Supply-chain audit

`cargo audit` is wired into CI (`.github/workflows/ci.yml` →
`daemon-audit` job, `--deny warnings`). At the time of this review the
dependency tree was clean — `cargo audit` exits 0 with no advisories
against any pinned version.

When `cargo audit` flags something in the future, fix the underlying
dependency. Allow-listing should happen only with explicit reasoning in
`pcside/daemon/audit.toml`.

## What we explicitly DID NOT do

To be honest about scope:

- **No paid third-party human audit.** This is a multi-agent review by
  the author, plus the third-party-crate / KAT cross-validation
  described above. Strong evidence; not the same as a Trail of Bits
  report.
- **No formal proof.** No symbolic execution, no model-checking, no
  proof of constant-time MAC compare under specific codegen.
- **No physical side-channel review of the ATmega328P.** Power
  analysis, EM emanations, and decap attacks remain explicitly
  out-of-scope per SPEC §13.1.
- **No PIN gate on the TPM seal yet.** PCR list parameterisation
  shipped; userAuth-bound seal (the canonical TPM "PCR + PIN" pattern)
  is a documented follow-up, not in this branch.

These limitations are listed in SPEC §13.1 (threat model out-of-scope
list), SPEC §13.10 (roadmap), and SPEC §13.12 (TPM follow-ups).
