Executive Summary

The r503-fpr daemon has been substantially hardened since the initial security pass (see commits S1–S9 + feat/tpm-seal + feat/crypto-posture-upgrade on main). The authenticated wire protocol (SipHash-2-4, monotonic counter, EEPROM-persisted replay defence) is implemented correctly on both sides, MAC comparison is constant-time, the D-Bus policy is deny-by-default + wheel-only, and the systemd unit applies an aggressive set of confinement directives (CapabilityBoundingSet=, PrivateNetwork=true, SystemCallFilter=@system-service, etc.).

This audit re-enters the codebase hostile, looking specifically for ways a local non-root user, a wheel-group user, or a process inheriting any of the device-node groups can (a) gain root via the fingerprint path, (b) impersonate another user to sudo / kscreenlocker / SDDM, or (c) recover the shared SipHash key. Physical access and pre-existing root are out of scope per the requesting policy.

Result. No critical privilege-escalation paths were found. One HIGH issue clusters around the serial-port access boundary: /dev/r503 inherits a permissive group from the OS default because the udev rule omits MODE= / GROUP=, and the daemon does not open the port with TIOCEXCL. Four MEDIUM findings are defense-in-depth gaps (unbounded RX buffer, missing file-mode/symlink checks on the host key file, no deny_unknown_fields on persisted JSON). The remainder are LOW or INFO — code-hygiene polish, parser edge cases, and clarifications where earlier auditors over-rated severity.

Scope & Threat Model

We assume an attacker who:

  • Has a local shell account on the host with no special privileges, OR a non-root account that happens to be in dialout / uucp / tty (whatever default group owns /dev/ttyACM* / /dev/ttyUSB* on the distro), OR a wheel-group account.
  • Can connect to the system D-Bus and call any method the policy permits.
  • Can open any device node the OS DAC mode permits.
  • Has read access to any world-readable file.

Explicitly out of scope per the brief: physical access to the device, pre-existing root, hardware lab attacks on the AVR, downstream compromise of PAM / kernel / systemd, and TPM side channels. We use the threat model from SPEC.md §13.1 as the baseline and stretch it where the host-side surface has been hardened past the original assumptions.

The attacker's goals we test against:

  • Enroll one of their own fingers under another user's identity (sudo / SDDM bypass).
  • Cause a verify to be misattributed to the wrong user.
  • Recover the 128-bit shared SipHash key from disk, memory, or wire.
  • Replay a captured OK match=… response to extend an auth window.
  • Deny service to the legitimate user (lock them out of fingerprint auth) and keep that within a stealth budget that doesn't trigger the password fallback being noticed.

Methodology

Six parallel auditors were dispatched against disjoint slices of the codebase, briefed adversarially and instructed to return structured findings. Their reports were then re-verified by direct inspection of the source — the highest-severity claims (a "CRITICAL" delete-no-claim path, a "HIGH" verify slot race, a "MEDIUM" TPM Owner-hierarchy bypass) were re-classified after reading the actual code: each turned out to be intentional design or guarded by a non-obvious gate the auditor missed. The audit therefore reports fewer issues than the raw subagent dump but each one survived a hostile second pass.

Detailed Findings

HIGH

H1 — /dev/r503 inherits permissive group + daemon does not TIOCEXCL the port

UDEV-SERIAL-COMPETE pcside/daemon/dist/70-r503.rules · pcside/daemon/src/sensor.rs:94
Description
The udev rules in 70-r503.rules create a /dev/r503 symlink with SYMLINK+="r503" but never set MODE=, OWNER=, or GROUP=. The resulting device permissions are inherited from the OS default for ttyACM* / ttyUSB* — on Fedora and Ubuntu that is 0660 root:dialout. Any account in the dialout group can open(2) the device and read/write bytes directly to the Nano. Compounding this, sensor.rs:94 opens the port with serialport::new(&path, DEFAULT_BAUD).timeout(…).open() with no exclusive flag — the underlying open(2) does not pass TIOCEXCL, so the daemon does not hold the port exclusively against a second opener.
Attack Vector
(a) Denial of service — paired Nano. A dialout user opens /dev/r503 and writes garbage bytes between the daemon's command and the Nano's response. The daemon's read sees corrupted frames, fails MAC verify, surfaces framing_rejected to PAM. fprintd retries; the attacker keeps writing. Fingerprint auth is denied; PAM falls back to password — the user is annoyed, opens KDE Settings, and finds no fault with the device. Persistent, low-noise DoS.

(b) Pair-race against a freshly wiped Nano. The reseal-tpm.sh recovery flow (SPEC §13.12) reflashes the Nano with r503fp_wipe, then reflashes the main firmware (briefly unpaired), then runs r503d --pair. During the window between "main firmware booted" and "daemon sends pair <hex>", an attacker watching udev events on /dev/r503 can write pair <attacker_hex>\n to the port themselves. The firmware's handlePair (r503fp.ino:223) accepts the attacker's key (it does not require a MAC in the unpaired state — TOFU bootstrap). The legitimate daemon then attempts to pair, the Nano responds ERR already_paired, --pair bails. The host's /etc/r503d/allow-pair gate guards the daemon's pairing decision but does not guard the Nano's. The Nano now holds an attacker-controlled key and the daemon has nothing matching it. Outcome: persistent denial of fingerprint auth until manual reflash-and-pair from a console session. This is not a privesc — the attacker can forge frames the legitimate daemon refuses to talk to, but cannot make the daemon return match=N for someone else's slot. Severity is therefore bounded at HIGH, not CRITICAL.

(c) Side-channel observation. A passive opener can read all framed traffic between daemon and Nano. The traffic is authenticated, not encrypted — counter values and matched slot numbers are visible in cleartext. SPEC §13.1 explicitly accepts that confidentiality is out of scope (templates never cross the wire), so this bullet is informational, but worth flagging here so it's accounted for.
Remediation
Two independent fixes, both cheap:
  1. Tighten 70-r503.rules so the device the symlink points at is owned by root with no group access. Add to every matching rule: OWNER="root" GROUP="root" MODE="0600". The r503d.service unit already grants the daemon (running as root) DeviceAllow=char-ttyACM rw + char-ttyUSB rw, so root-only DAC is sufficient.
  2. Open the port exclusively in the daemon. The serialport crate exposes a Unix-only exclusive(true) builder method backed by TIOCEXCL. Add .exclusive(true) to the chain at sensor.rs:94 and to the Link::open path in pairing.rs. After TIOCEXCL any second open(2) from a non-root user returns EBUSY, which closes both the DoS angle and the pair-race even on hosts with a more permissive distro default than the udev fix anticipated.
MEDIUM

M1 — Unbounded RX buffer in read_line()

SENSOR-UNBOUNDED-RXBUF pcside/daemon/src/sensor.rs:234–260
Description
R503::read_line() grows self.rx_buf via extend_from_slice(&buf[..n]) until it sees a \n or the deadline expires. There is no hard cap on the line length. If the byte stream contains no newline within the deadline, the buffer still keeps growing each loop iteration (only the deadline check exits — not a length check). Combined with H1, a co-opener can stream arbitrary bytes; alternatively a defective Nano (or a Nano in the middle of a bootloader stuck state) can saturate memory.
Attack Vector
Local user in dialout opens /dev/r503 and writes a long stream of non-\n bytes faster than the daemon's deadline elapses. The daemon's rx_buf grows past the 1024-byte initial capacity, allocating each time the backing Vec doubles. Memory pressure spikes the daemon and, under MemoryDenyWriteExecute=true + cgroup memory limits, can trigger OOM and service restart. Repeated, this is a stealth DoS.
Remediation
Add a constant — e.g. const MAX_LINE_LEN: usize = 4096; — and check self.rx_buf.len() > MAX_LINE_LEN on every iteration of the inner read loop. On overflow, drop the buffer, flush the kernel input queue (self.port.clear(ClearBuffer::Input)), and return SensorError::Protocol("line too long"). 4 KiB is ~50× the spec frame size (longest realistic framed line is ~120 bytes) and well below the firmware's inbuf > 128 cap mirror.
MEDIUM

M2 — load_key follows symlinks and accepts any file mode

KEYSTORE-NO-NOFOLLOW-NO-MODE-CHECK pcside/daemon/src/keystore.rs:54–66, 73–87
Description
keystore::load_key() uses fs::read_to_string(KEY_PATH) which follows symlinks via open(2) with no O_NOFOLLOW, and the loader never inspects fs::metadata(KEY_PATH).permissions().mode() to verify the live key is 0600 and the backup is 0400. The key directory /var/lib/r503d is created 0700 root, so a non-root attacker can't pre-stage a symlink there — but the protection is entirely "trust the directory permissions" with zero belt-and-suspenders inside the load path.
Attack Vector
Two narrow but real paths:
  1. A compromised package post-install hook, an unrelated root-as-user service with a path-traversal bug, or an installer race (the install.sh flow creates the directory) can produce a symlink at /var/lib/r503d/key pointing at /etc/shadow. The daemon reads the target file into a Zeroizing<String>, fails parse_key_hex, but the file's bytes have already passed through process memory. A subsequent crash producing a coredump exposes the bytes.
  2. A non-key-but-shaped-like-a-key file (32 hex chars) written by any process briefly running as root with a write to /var/lib/r503d/ can land at /var/lib/r503d/key.bak with mode 0644. The loader's fallback path accepts it.
Neither yields privesc on its own, but both undercut the assumption that an unsealed plaintext key is only reachable by root.
Remediation
Replace fs::read_to_string(p) with a guarded open: stat the path with fs::symlink_metadata (does NOT follow symlinks), refuse if file_type().is_symlink(), refuse if uid != 0, refuse if mode & 0o077 != 0 (i.e. any group/other access). Then open with OpenOptions::new().read(true).custom_flags(O_NOFOLLOW).open(p) and read the bytes. Mirror the same checks for KEY_BAK_PATH with a stricter 0o177 mask.
MEDIUM

M3 — Persisted JSON files accept unknown fields and don't verify file mode

JSON-LOOSE-LOAD pcside/daemon/src/storage.rs · state.rs
Description
Both StorageFile (the on-disk container behind users.json) and State (the counter file at state.json) deserialize with serde_json::from_slice and the default #[derive(Deserialize)] — meaning unknown fields are silently ignored and no schema-drift error is raised. Neither loader checks the file's mode on load. Combined, the daemon accepts a users.json with unknown keys (defense-in-depth break: a hostile root, or a future buggy migration, could leave undetected drift in the file) and an attacker who can briefly relax users.json to 0644 can read who is enrolled and at which slot — useful for reconnaissance ahead of a templated-finger attack.
Attack Vector
Low-direct-impact, but the cost of #[serde(deny_unknown_fields)] is negative (one annotation per struct) and the cost of a mode check is one fs::metadata().permissions().mode() == 0o600 branch. The status quo means a stale or maliciously-modified file is silently accepted — which has bitten every system that decided "we don't need deny_unknown_fields here" eventually.
Remediation
Annotate the on-disk structs: 
#[derive(Serialize, Deserialize)]
#[serde(deny_unknown_fields)]
struct StorageFile { … }

#[derive(Serialize, Deserialize)]
#[serde(deny_unknown_fields)]
pub struct State { … }
And in both load() functions, before parsing, check the mode with let mode = fs::metadata(p)?.permissions().mode() & 0o777; if mode != 0o600 { bail!("insecure mode {:o} on {p}", mode); }. Pair with the symlink check from M2.

M4 — Firmware silently truncates the inner command body to 96 bytes after MAC verify

FW-DISPATCH-TRUNCATION firmware/r503fp/r503fp.ino:331–336
Description
After verify_command_frame() returns FRAME_OK with inner_len > cap (where cap = 95), the firmware copies the first 95 bytes into ibuf and dispatches the truncated command: 
char ibuf[96];
size_t cap = sizeof(ibuf) - 1;
size_t n = (inner_len < cap) ? inner_len : cap;
memcpy(ibuf, inner, n);
ibuf[n] = 0;
dispatch(String(ibuf));
The MAC has already been verified over the full inner_len bytes, so the truncated form has no integrity gate. In the current command set the longest body is ~10 bytes (verify 199), so this is far from any practical attacker craft, but the silent failure mode means a future command added with a longer payload (e.g. a set-policy <json> admin verb) could be exploitable on day one.
Attack Vector
Forward-looking. An attacker who controls the host (which is the already-out-of-scope case for the current command set) crafts a long valid frame whose MAC-verified prefix is one command and whose dropped suffix changes meaning (e.g. enroll 5 confirm → enroll 5). For the current firmware command set this is not exploitable, so the severity is MEDIUM and not HIGH — but the codepath is the kind that ages badly.
Remediation
Reject overlong inner bodies before dispatch instead of truncating. Add to process_line() after the MAC + counter checks pass: 
if (inner_len >= sizeof(ibuf)) {
  Serial.println(F("ERR frame_body_too_long"));
  return;
}
The buffer size 96 is then a hard reject, not a silent truncation, and any future command-set addition that needs more space must explicitly bump the buffer.
MEDIUM

M5 — reseal-tpm.sh resolves arduino-cli and r503d via $PATH with no hardening

RESEAL-PATH-SOFT pcside/daemon/dist/reseal-tpm.sh
Description
reseal-tpm.sh uses command -v arduino-cli and command -v r503d to discover binaries (with a fallback to $SUDO_USER's home for the Arduino tooling). The script does set -euo pipefail and is invoked as sudo bash reseal-tpm.sh, so sudo's default secure_path provides most of the protection — but the script itself does not export PATH=/usr/sbin:/usr/bin:/sbin:/bin at the top. A sudoers configuration with env_keep += PATH (more common than it should be) gives any user able to run the script via sudo the ability to drop a malicious arduino-cli in an early $PATH entry and have it execute as root.
Attack Vector
An admin's sudoers rule grants NOPASSWD: /usr/local/sbin/reseal-tpm.sh to the wheel group and leaves env_keep+=PATH intact. Attacker (wheel) runs PATH=/tmp/evil:$PATH sudo /usr/local/sbin/reseal-tpm.sh with /tmp/evil/arduino-cli chmod +x. Script executes the attacker's binary as root. Privesc.
Remediation
Pin PATH at the top of the script, before any function definitions: 
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
export PATH
Optionally also use /usr/bin/env -i in the shebang or call out absolute paths for the few external tools used (systemctl, install, arduino-cli). The current shebang is #!/usr/bin/env bash, which is fine, but combine it with the explicit PATH reset.
LOW

L1 — verify_start reads user_slots after dropping the state lock

VERIFY-SLOT-RACE pcside/daemon/src/dbus_iface.rs:470–509
Description
The action token is created inside a state.lock() scope at line 471–479, then the lock is dropped, and self.storage.get_user_slots(&user) is called at line 485 to snapshot the expected slot set. Between those points, another D-Bus call sharing the same claim could in principle race a mutation. In practice every mutating method that touches the user's slots either (a) requires ensure_claim to match the same sender (so cross-sender races are blocked at the gate), or (b) goes through delete_enrolled_fingers, which is polkit-gated and denied by default. Same-sender self-races are possible — the user can call DeleteEnrolledFinger concurrently with their own VerifyStart — but the worst case is that the verify naturally fails (the template is gone from sensor flash too) without any privilege change.
Attack Vector
Bounded by ensure_claim. No realised privesc; included as a defense-in-depth hygiene note.
Remediation
Snapshot user_slots while still holding the state lock at line 471–479, or hold a per-action read-lock on the user's slot set for the lifetime of the verify task. The cleanest fix is to capture the slots before spawning the verify task and pass the snapshot in by value, so the spawned task never re-reads storage.
LOW

L2 — MAC input formatted into heap String without Zeroizing

MAC-INPUT-NOT-ZEROIZED pcside/daemon/src/framing.rs:43, 65, 87, 119
Description
Each MAC compute path allocates a String on the heap to hold the domain-separated MAC input ("CMD <ctr> <cmd>" or "RSP <ctr> <seq> <body>") and lets it drop normally. The buffer is not wrapped in Zeroizing; on drop the allocator may reuse the slab and a heap-scanning attacker (or a future bug producing a coredump) can observe recent MAC inputs. The inputs are not themselves secret — they contain only command text, counter, seq, body — but the discipline of wrapping anything that transits a key-derivation primitive in Zeroizing is what's missing.
Attack Vector
Negligible. The key itself is already in a Zeroizing<[u8;16]> and the MAC input is public per the protocol. Worth fixing for consistency with the rest of the codebase, not for direct exploitability.
Remediation
Replace each let mac_input = format!(…); with let mut mac_input = Zeroizing::new(String::with_capacity(N)); use std::fmt::Write; write!(&mut *mac_input, …)?; Or — slightly more idiomatic — build a stack [u8; 256] via arrayvec and skip the heap entirely.
LOW

L3 — allow-pair gate is a benign TOCTOU on a root-only directory

ALLOW-PAIR-TOCTOU pcside/daemon/src/pairing.rs:192–228
Description
run_pair calls keystore::allow_pair_present() at line 192, then continues to generate_key at line 219 and remove_allow_pair at line 228. The check and the remove are not atomic, and the file lives in /etc/r503d/ which is root-owned. A non-root attacker cannot influence the check. A root attacker can — but root is out of scope per the threat model. Listing this for completeness because earlier auditors flagged it.
Attack Vector
None within the threat model.
Remediation
None required. If you want a belt-and-suspenders fix anyway, move remove_allow_pair() to immediately follow the check, then generate the key — this would make the file removal a hard pre-commit rather than a tail action, and cost nothing.
LOW

L4 — Input-buffer overflow gate is off-by-one

FW-INBUF-OFF-BY-ONE firmware/r503fp/r503fp.ino:432
Description
The firmware's per-line cap is implemented as if (inbuf.length() > 128) { /* reject */ }. This accepts a 129-byte payload by the time the check fires (the 129th byte has already been appended to inbuf). Because inbuf is a dynamic Arduino String, there is no stack smash and no exploitable consequence — but the documented invariant ("lines are capped at 128 bytes") is silently violated by one byte. Flag for hygiene.
Attack Vector
None.
Remediation
Change to >= 128, or define a named constant kInbufMax = 128 and check before append.
LOW

L5 — Username from getpwuid_r not validated against an allow-list

USERNAME-NO-ALLOWLIST pcside/daemon/src/dbus_iface.rs:pwd_lookup
Description
pwd_lookup returns the username string verbatim from getpwuid_r(3). Modern Linux useradd rejects usernames with slashes, control chars, etc., but the daemon does not enforce its own allow-list. A username containing path separators or control characters would survive the lookup and become a HashMap key in users.json. Direct exploitability requires a future code path that interpolates the username into a file path or a shell argument — none exists today.
Attack Vector
Forward-looking. No realised exploit on the current code surface.
Remediation
Add a one-line validator at the boundary in auth::resolve_caller: reject usernames not matching ^[a-zA-Z_][a-zA-Z0-9._-]{0,31}$. Logs the rejected name and returns FprintError::Internal.
LOW

L6 — Polkit decision is not cached

POLKIT-NO-CACHE pcside/daemon/src/auth.rs:81–101
Description
Every cross-user request (i.e. every call where the requested username differs from the caller and the caller isn't uid 0) makes a fresh check_authorization RPC to polkitd. There is no per-caller cache of the result. A wheel-group user issuing rapid Claim "victim" calls produces a 1:1 RPC fan-out to polkit. Not a privesc, but a tiny amplification factor that could contribute to a polkit-side DoS if the system is already stressed.
Attack Vector
Negligible on a workstation; modestly useful as part of a wider DoS chain.
Remediation
Optional. If implemented: a small Mutex<HashMap<(u32, &'static str), (Instant, bool)>> keyed on (caller_pid, action_id) with a 1-second TTL would be sufficient. The current behavior (always-fresh polkit) is the safer default and the perf hit is not measurable on real workloads — leave it unless monitoring flags it.
LOW

L7 — install.sh co-tenants with upstream fprintd

INSTALL-COEXIST-NOISE pcside/daemon/dist/install.sh
Description
The install script overwrites /usr/share/dbus-1/system-services/net.reactivated.Fprint.service and the matching D-Bus policy file — both upstream fprintd paths. If both packages are installed, the last one wins and the loser silently breaks. Mitigation in the systemd unit is Conflicts=fprintd.service (good), but the .service / .conf replacement on disk is irreversible by the uninstall script (which removes our files, leaving fprintd without its own). This is operational, not security — but a user-confused-by-bricked-fprintd may reach for "fix it fast" tactics that re-open attack surface.
Attack Vector
None within the threat model.
Remediation
Make install.sh back up the upstream files before overwrite (mv -n /usr/share/dbus-1/system-services/net.reactivated.Fprint.service{,.fprintd.bak}) and have uninstall.sh restore them if present.

I1 — RECLASSIFIED — DeleteEnrolledFingers without claim is polkit-gated by design

DEL-NO-CLAIM-BY-DESIGN pcside/daemon/src/dbus_iface.rs:335–348
Note
An auditor flagged the legacy DeleteEnrolledFingers(username) method as a CRITICAL privesc because it does not call ensure_claim. Direct re-read of lines 335–348 confirms it calls auth::authorize_username, which routes cross-user requests through polkit's net.reactivated.fprint.device.setusername action. The shipped polkit policy denies that action by default for every session class (allow_any=no, allow_inactive=no, allow_active=no), and the comment at lines 341–344 explicitly explains the design choice — fprintd parity via polkit, with a claim-gated DeleteEnrolledFingers2 alongside for tighter flows. Severity is INFO; no action recommended.
INFO

I2 — RECLASSIFIED — Owner hierarchy with empty auth does not bypass the PCR seal

TPM-OWNER-NO-BYPASS pcside/daemon/src/tpm.rs
Note
An auditor flagged the use of Hierarchy::Owner with an unset Owner auth value as a path that bypasses the PCR policy. This is incorrect. The sealed object is created with userWithAuth = false and adminWithPolicy = true, and the unseal path supplies a PolicyPCR session — TPM2_Unseal will fail unless the session's digest matches the policy baked into the object at seal time. The Owner hierarchy's auth governs the parent key's lifecycle (create / evict), not the sealed child's unseal authorization. The seal stands regardless of Owner-auth setting. No action needed.
INFO

I3 — RECLASSIFIED — Subject::new_for_owner(pid, None, None) is the safe form

POLKIT-START-TIME-SAFE pcside/daemon/src/auth.rs:89
Note
An auditor speculated that passing None for the unix-process start_time to zbus_polkit::Subject::new_for_owner would let polkitd authorize on pid alone and be vulnerable to pid reuse. In practice polkitd itself reads start_time from /proc/<pid>/stat on receipt of the call when the caller didn't supply one, so the (pid, start_time) tuple is bound on the polkit side. The comment at auth.rs:84–88 documents this. INFO only; no change.

I4 — Confidence score on the authenticated channel does not leak key material

FW-CONFIDENCE-OK firmware/r503fp/r503fp.ino:168–187
Note
The R503's per-match confidence score is returned in the framed OK match=N confidence=M response. It correlates with finger image clarity (texture, pressure, area), not with the SipHash key. SPEC §13.1 acknowledges that confidentiality is out of scope. No issue.

What's Solid

The codebase has clear defensive design and the hardening commits (S1–S9 + TPM-seal + crypto-posture) show up in the right places. Specifically:

Authentication of the wire protocol

SipHash-2-4 implementation matches Aumasson reference vectors on both sides; the firmware runs a boot-time KAT self-test (r503fp.ino:361–392) that halts the framed channel if the primitive diverges; the host-side cross-validates against the third-party siphasher crate over 1024 random inputs. MAC comparison is constant-time on both sides (subtle::ConstantTimeEq on Rust, unconditional XOR-OR accumulator on AVR). Domain separation between CMD and RSP MAC inputs is enforced and verified by tests. Counter persistence ordering is correct (persist before send on host, persist before handler on firmware) — both designed to survive a crash without enabling replay.

Caller authorization

auth::authorize_username is the single chokepoint for every username-accepting D-Bus method. Empty-string → caller, self-request fast path, uid 0 (PAM) fast path, cross-user via polkit with no AllowUserInteraction (so an SSH attacker can't pop a dialog on the active session). The shipped polkit policy defaults every session class to no, so the cross-user gate is closed unless an admin explicitly grants it via /etc/polkit-1/rules.d/. The D-Bus policy is deny-by-default with explicit allows for wheel + root only on the specific interfaces — stricter than upstream fprintd.

Systemd confinement

r503d.service applies a wide hardening set: ProtectSystem=strict, ProtectHome=true, PrivateTmp=true, PrivateNetwork=true, NoNewPrivileges=true, RestrictAddressFamilies=AF_UNIX, SystemCallFilter=@system-service, MemoryDenyWriteExecute=true, LockPersonality=true, RestrictNamespaces=true, RestrictSUIDSGID=true, CapabilityBoundingSet= (empty), DevicePolicy=closed + explicit DeviceAllow for char-ttyACM / char-ttyUSB / /dev/tpmrm0, ProtectClock=true, ProtectHostname=true, ProtectKernelLogs=true, ProtectKernelTunables=true, ProtectKernelModules=true, ProtectControlGroups=true. A successful exploit of r503d would land in a process that can do almost nothing other than touch /var/lib/r503d and the serial char device.

Key handling

The 128-bit key is wrapped in Zeroizing<[u8;16]> from generation (getrandom::fill) through use and into save. Hex form is also Zeroizing<String>. Plaintext files are written atomically (tmp + fsync + rename) at 0600 / 0400, and deletion shreds with 0xFF before unlink. TPM seal binds to PCR7 by default, with operator-selectable extra PCRs; on a PCR mismatch the daemon refuses to start rather than falling back to plaintext, and the reseal ceremony preserves enrolled templates. No plaintext-key fallback exists when a sealed blob is present — that path is removed so it can't be coerced by an attacker who manages to delete the sealed file.

Replay defence

Wear-leveled EEPROM counter ring (16 cells × CRC-16-CCITT) holds the Nano's last_seen across reboots; host's next_cmd_counter is persisted atomically before each send. Pairing is TOFU with an explicit host-side opt-in (/etc/r503d/allow-pair) so an attacker plugging in a hostile Nano cannot auto-pair. The unpair path is framed (proves key knowledge); the wipe-and-reflash path is the documented recovery for lost-key scenarios.

Conclusion

The r503-fpr stack does what a hobbyist security-conscious project rarely does: it has an explicit threat model, hardening that addresses that model layer by layer, and tests (KAT, fuzz, cross-impl, end-to-end PAM regression) that prove the hardening still works after each refactor. No critical privilege-escalation paths exist within the documented threat model and the security boundaries hold against the audit's stretch goals (wheel user / dialout user / passive observer on the wire).

The single HIGH finding is the implicit one: the daemon assumes it has exclusive access to /dev/r503 by virtue of the udev rule, but the udev rule never makes that assumption true and the daemon's open call doesn't enforce it. Two ~5-line changes close it. The remaining findings are defense-in-depth polish that bring the codebase to a posture consistent with the seriousness of its threat model documentation.
