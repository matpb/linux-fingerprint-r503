Executive Summary

All 18 findings were validated by direct source inspection. 13 of 14 actionable findings were confirmed accurate and have a fix committed (10 this session — M2–M5, L1, L2, L4, L5 — plus H1 and M1 which landed earlier on this branch). One finding (L7) did not validate: the install script writes to override paths (/usr/local/share, /etc), never to the upstream fprintd /usr/share files, and the uninstaller removes exactly those overrides — so the claimed "irreversible overwrite of upstream fprintd" does not occur. Two LOW findings (L3, L6) were confirmed accurate as descriptions but deliberately not "fixed": the audit itself recommends no change for L3 (benign TOCTOU on a root-only directory) and explicitly recommends against the L6 change (caching a polkit decision is a security downgrade, not an upgrade). The five reclassified INFO items (I1–I5) were re-confirmed as non-issues.

Every Rust change was gated on cargo build --release + cargo test --release (92 tests, all green); every firmware change on arduino-cli compile (atmega328, 70% flash). No fix changes a happy-path behaviour — they add belt-and-suspenders that the existing, correctly-permissioned files already satisfy.

Detailed Validation

HIGH

H1 — udev permissive group + serial exclusivity

70-r503.rules · sensor.rs · pairing.rs
Validation
udev half: CONFIRMED. 70-r503.rules had no MODE=/OWNER=/GROUP= on any of its five SUBSYSTEM rules, so /dev/r503 inherited the distro default (0660 root:dialout). TIOCEXCL half: REVISED. The audit claimed the daemon's serialport::new(...).open() does not pass TIOCEXCL. That is incorrect — serialport 4.x defaults exclusive: true on POSIX, so the port was already held exclusively.
Fix 1d97b7b
Added OWNER="root" GROUP="root" MODE="0600" to every matching rule (root-only DAC; the systemd unit already grants the daemon the device via DeviceAllow). Pinned .exclusive(true) explicitly at both open sites so a future crate-default flip can't silently downgrade, and the posture is visible at the call site.
MEDIUM

M1 — Unbounded RX buffer in read_line()

sensor.rs
Validation
CONFIRMED. The inner read loop grew rx_buf via extend_from_slice with only a deadline exit — no length cap.
Fix e6203c7
Added const MAX_LINE_LEN = 4096 (~30× the longest realistic framed line) and a per-iteration cap that clears the buffer, flushes the kernel input queue, and returns SensorError::Protocol.
MEDIUM

M2 — load_key follows symlinks & ignores mode

keystore.rs
Validation
CONFIRMED. load_key() read both key files with a bare fs::read_to_string (follows symlinks, no O_NOFOLLOW) and never inspected owner or mode. The 0700 root directory makes pre-staging hard, but the load path trusted it entirely.
Fix 1370571
New read_key_file_guarded(): symlink_metadata + reject symlink, reject owner ≠ 0, reject access bits outside the allow-mask (0o077 live / 0o177 backup), then open with O_NOFOLLOW to defeat a post-stat TOCTOU swap. Failed checks log and fall through exactly as the old None path did.
MEDIUM

M3 — Loose JSON loaders, no mode check

storage.rs · state.rs
Validation
CONFIRMED. StorageFile and State used the default derive (unknown fields silently ignored); neither open()/load() checked the file mode.
Fix 8e4230f
#[serde(deny_unknown_fields)] on both structs; both loaders now refuse a file whose mode isn't exactly 0600. Both save() paths already write 0600 (state since inception, users.json since S3), so the happy path is unchanged. New StorageError::InsecureMode.
MEDIUM

M4 — Firmware silently truncates inner body

r503fp.ino
Validation
CONFIRMED at lines 331–336. After MAC verify over the full inner_len, the body was memcpy-clipped to 95 bytes and dispatched — no integrity gate on the truncated form. Not reachable for the current ~10-byte command set, but a latent footgun for a longer future verb.
Fix baab047
Reject inner_len >= sizeof(ibuf) with ERR frame_body_too_long, placed after the counter commit (no replay) and before g_out swings to the framer (error goes out unframed).
MEDIUM

M5 — reseal-tpm.sh resolves binaries via $PATH

dist/reseal-tpm.sh
Validation
CONFIRMED. The script does set -euo pipefail but never resets PATH; binaries are found via command -v. A sudoers env_keep += PATH turns that into root-level command injection.
Fix e8865ea
Pinned PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin right after set -euo pipefail. No regression: r503d lives in /usr/local/bin, and the arduino-cli lookup keeps its ARDUINO_CLI/$SUDO_USER fallbacks.
LOW

L1 — verify_start reads slots after dropping the lock

dbus_iface.rs
Validation
CONFIRMED. Token minted under the state lock, lock dropped, then get_user_slots read in a fresh acquisition. A same-sender DeleteEnrolledFinger (claim-gated, not action-gated) can mutate the set in the gap. Bounded by ensure_claim — no privesc; worst case a verify that naturally fails.
Fix 5b56b51
Mint the token and read get_user_slots under one lock acquisition. get_user_slots only touches the independent storage RwLock, so holding the state mutex across it is deadlock-free.
LOW

L2 — MAC input String not Zeroizing

framing.rs
Validation
CONFIRMED at the four MAC-compute sites. Exploitability is negligible — the MAC input is public per the protocol — so this is a consistency/hygiene gap, not a confidentiality one.
Fix 03336d4
Wrapped each format!() in Zeroizing::new(...); the temporary is moved (no extra alloc), .as_bytes() still works via Deref, buffer scrubs on drop.
LOW

L3 — allow-pair TOCTOU

pairing.rs
Validation
CONFIRMED as described: allow_pair_present() and remove_allow_pair() aren't atomic. The file lives in /etc/r503d/ (root-owned), so a non-root attacker can't influence the check; a root attacker is out of scope. No gap within the threat model.
Disposition
No-fix by design. The report's own remediation is "None required." Note: the gate is already closed before the key is sent (pairing.rs:233, the S2 hardening), so the meaningful ordering concern was handled in a prior pass. Reordering further would add churn for zero security gain.
LOW

L4 — inbuf cap off-by-one

r503fp.ino
Validation
CONFIRMED at line 432: > 128 admits a 129th byte before firing. No exploitable consequence (dynamic String), but the documented 128-byte invariant was violated by one.
Fix 9d712c9
> → >=. Real frames are ~50 bytes, so no regression.
LOW

L5 — Username not validated

auth.rs · dbus_iface.rs
Validation
CONFIRMED. resolve_caller returned the getpwuid_r username verbatim. Forward-looking only — nothing interpolates it into a path or shell today.
Fix 0b92257
validate_username() at the boundary rejects empty, ./.., path separators, and NUL/control bytes. Deliberate deviation: the report sketched a tight ^[a-zA-Z_][...]{0,31}$ allow-list — that would wrongly bounce locale/site-specific accounts a working getpwuid_r legitimately returns, i.e. a regression. We reject only the genuinely path-dangerous shapes (all of which useradd already forbids), fully closing the named gap with zero risk to real accounts. Unit-tested both ways.
LOW

L6 — Polkit decision not cached

auth.rs
Validation
CONFIRMED. Every cross-user request makes a fresh check_authorization RPC; there is no cache.
Disposition
No-fix by design. The report explicitly recommends against implementing this: "The current behavior (always-fresh polkit) is the safer default… leave it unless monitoring flags it." Caching an authorization decision introduces staleness risk — it is a security downgrade, so implementing it would be the regression the task forbids.
LOW

L7 — install.sh "overwrites upstream fprintd"

dist/install.sh · dist/uninstall.sh
Validation
DID NOT VALIDATE. The claim is that install.sh overwrites the upstream fprintd files at /usr/share/dbus-1/system-services/net.reactivated.Fprint.service and the matching policy, irreversibly. Reading the script:
  • install.sh:54–57 writes the D-Bus service activation file to /usr/local/share/dbus-1/system-services/ — an override directory that shadows, never the upstream /usr/share path.
  • install.sh:64–66 writes the bus policy to /etc/dbus-1/system.d/ — the admin override dir, not upstream's /usr/share/dbus-1/system.d/.
  • uninstall.sh:20,24 removes exactly those override files, so the untouched upstream fprintd files are restored automatically; the script also systemctl unmaskes and restarts fprintd.
No upstream file is ever overwritten, and the revert is clean. The finding's premise is inaccurate.
Disposition
No fix. (If anything, the existing co-tenancy design is the correct one — override dirs + Conflicts=fprintd.service.)

I1 — DeleteEnrolledFingers without claim

dbus_iface.rs:335–348
Re-confirmed
The legacy delete_enrolled_fingers calls auth::authorize_username (line 346), routing cross-user requests through the polkit setusername action, which the shipped policy denies by default. Claim-gated DeleteEnrolledFingers2 exists alongside. No action.
INFO

I2 — TPM Owner hierarchy "bypasses" PCR seal

tpm.rs
Re-confirmed
sealed_object_template builds the object with with_admin_with_policy(true) + with_user_with_auth(false) and bakes the PolicyPCR digest into auth_policy; unseal_key supplies a PolicyPCR session. The Owner hierarchy governs only the deterministic SRK parent's lifecycle, not the sealed child's unseal authorization. No bypass. No action.
INFO

I3 — Subject::new_for_owner(pid, None, None)

auth.rs:89
Re-confirmed
polkitd reads start_time from /proc/<pid>/stat when the caller passes None, so the (pid, start_time) tuple is bound on the polkit side — not pid-reuse racy. Documented in the comment at auth.rs:84–88. No action.

r503fp.ino:168–187
Re-confirmed
The per-match confidence in OK match=N confidence=M correlates with finger-image clarity, not the SipHash key, and rides the authenticated channel. SPEC §13.1 puts confidentiality out of scope. No action.
